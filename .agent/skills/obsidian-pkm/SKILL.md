# obsidian-pkm

Skill นี้ออกแบบมาเพื่อให้ AI และผู้ใช้งานทำงานร่วมกันในรูปแบบฉบับของ **Obsidian Personal Knowledge Management (PKM)** โดยมีเป้าหมายสูงสุดคือ **"ลดการสิ้นเปลือง Token และรักษาความสะอาดของ Context Window"**

## 📂 1. โครงสร้างโฟลเดอร์หลัก (The Vault Architecture)

โฟลเดอร์ `docs/` ในโปรเจกต์นี้ทำงานเป็น Obsidian Vault AI จะต้องเคารพและรักษาโครงสร้างเหล่านี้เสมอ:
- `docs/00_Dashboard.md` (MOC): ศูนย์กลางและสารบัญหลัก (อ่านที่นี่ถ้าหลงทิศทาง)
- `docs/01_Active_Tasks/`: พื้นที่สำหรับ "งานที่กำลังทำอยู่ตอนนี้" เท่านั้น
- `docs/02_Architecture/`: ข้อมูลโครงสร้างระบบ เช่น Database Schema หรือหน้าจอ UI (ควรแตกย่อยเป็นส่วนๆ)
- `docs/03_Prompts_Rules/`: เป็นแหล่งรวม Template การสั่งงาน AI หรือสารานุกรม
- `docs/99_Archive/`: ถังสำหรับเก็บงานที่ "เสร็จแล้ว" หรือ "รอรื้อฟื้น" ห้าม AI ยุ่งกับโฟลเดอร์นี้เพื่อประหยัด Token (เว้นแต่ถูกสั่งให้ค้นหา)

## 🧠 2. กฎการประหยัด Token และ Atomic Notes (CRITICAL)

- **การซอยไฟล์ (Atomicity):** ห้าม AI สร้างหรือตอบสนองเป็นไฟล์ Markdown (MD) เดี่ยวๆ ที่มีความยาวเกิน 150-200 บรรทัด. หากระบบยาวเกินไป จะต้องทำการซอยไฟล์ (Split Files) แบ่งออกเป็นกระบวนการย่อยๆ
- **การใช้ Wiki-links:** เมื่อมีการพูดถึงส่วนอื่น ให้ใช้โครงสร้างแบบ Obsidian คือ `[[ชื่อไฟล์หรือหัวข้อ]]` เพื่อให้ผู้ใช้สามารถกดคลิกไปยังไฟล์นั้นได้โดยตรง และเป็นการบอก AI ถึงความเชื่อมโยง

## 🔄 3. กฎเมื่อปฏิบัติงานเสร็จสิ้น (The Archive Rule)

> [!important] 
> เมื่อฟีเจอร์ใดฟีเจอร์หนึ่ง หรือแผนงานใน `docs/01_Active_Tasks/` มีสถานะเป็น **완료 (Completed)** AI มีหน้าที่ **บังคับ** แจ้งเบาะแสอัปเดตไฟล์ให้ผู้ใช้ทราบ ดังนี้:
> *"งานนี้เสร็จสิ้นแล้ว เพื่อป้องกันการเปลือง Token ในอนาคต ให้ย้ายไฟล์นี้ไปที่ `99_Archive/` ได้เลยครับ"* 

## ✍️ 4. ฟอร์แมตการสร้าง Note ใหม่ของ Obsidian

ทุกครั้งที่มีคำสั่งให้ AI สร้างไฟล์ Markdown เอกสารใหม่ใน `docs/` ไฟล์เหล่านั้นบังคับต้องมีคุณสมบัติ:
1. **Frontmatter:** หัวไฟล์มี YAML (`---`) ระบุ Title, ขั้นตอน และ Tags
2. **Callouts:** ใช้ GitHub Callouts (`> [!info]`, `> [!warning]`) ในการเน้นยำข้อความ ไม่ใช้แค่ตัวหนา
3. **Checklist:** หากมีสิ่งที่ต้องทำ ใช้รูปแบบ `- [ ]` เสมอ
